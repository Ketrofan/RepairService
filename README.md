Repair Service — “Заявки в ремонтную службу” (Laravel 12 + Docker Compose)
==========================================================================

Небольшое веб-приложение для приёма и обработки заявок в ремонтную службу.

--------------------------------------------------------------------------
Стек
--------------------------------------------------------------------------
- Laravel 12 (PHP 8.3)
- MySQL 8.0 (Docker Compose)
- UI: Blade (без SPA)
- Тесты: PHPUnit / Laravel Feature tests

Требования
- Git
- Docker + Docker Compose (docker compose ...)

--------------------------------------------------------------------------
Быстрый старт (Docker Compose)
--------------------------------------------------------------------------
Команды выполняются в терминале в корне проекта (там, где docker-compose.yml).

0) Склонировать репозиторий
   git clone <repo>
   cd <repo>

1) Поднять контейнеры
```bash
   docker compose up -d --build
```
   Если сборка падает из-за Docker Hub (например TLS handshake timeout), попробуй:
```bash
   export DOCKER_CLIENT_TIMEOUT=300
   export COMPOSE_HTTP_TIMEOUT=300
   docker compose build --no-cache
   docker compose up -d
```
2) Дождаться готовности приложения (первый запуск)
   При первом запуске контейнер app может устанавливать зависимости Composer (создаёт vendor/).
   Посмотреть прогресс:
``` bash
   docker compose logs -f app
```
   Когда в логах появится строка:
   READY: You can now run migrations
   — можно запускать миграции.

3) Подготовить БД (миграции + сиды)
```bash
   docker compose exec app php artisan migrate:fresh --seed
```
4) Открыть приложение
   http://localhost:8000

--------------------------------------------------------------------------
Модель заявки (Request)
--------------------------------------------------------------------------
Таблица: requests (snake_case)

Поля:
- client_name — обязательно
- phone — обязательно, формат +7XXXXXXXXXX (строго +7 + 10 цифр)
- address — обязательно
- problem_text — обязательно
- status — одно из: new | assigned | in_progress | done | canceled
- assigned_master_id — nullable
- created_at, updated_at

--------------------------------------------------------------------------
Экраны (обязательные по ТЗ)
--------------------------------------------------------------------------
Открывать в браузере:

- Создание заявки:
  GET /requests/create → http://localhost:8000/requests/create

- Панель диспетчера (список/фильтр/назначить/отменить):
  GET /dispatcher/requests → http://localhost:8000/dispatcher/requests

- Панель мастера (список назначенных/взять в работу/завершить):
  GET /master/requests → http://localhost:8000/master/requests

--------------------------------------------------------------------------
Роли и доступ
--------------------------------------------------------------------------
Dispatcher:
- видит список всех заявок
- фильтрует по статусу
- назначает мастера: new → assigned
- отменяет заявку: * → canceled (кроме done/canceled)

Master:
- видит заявки, назначенные на себя
- “Взять в работу”: assigned → in_progress
- “Завершить”: in_progress → done

--------------------------------------------------------------------------
Защита от гонки “Взять в работу”
--------------------------------------------------------------------------
Действие “Взять в работу” реализовано безопасно: атомарным UPDATE с условием
по текущему статусу и назначенному мастеру.

Корректное поведение:
- один параллельный запрос успешен (HTTP 200),
- второй получает отказ (HTTP 409) и понятное сообщение (“уже взято / недоступно”).

--------------------------------------------------------------------------
Проверка гонки (реальная) — race_test.sh
--------------------------------------------------------------------------
Подготовка в UI:
1) Войти диспетчером: dispatcher@example.com / password
2) Создать заявку: /requests/create
3) В /dispatcher/requests назначить мастера Master #1
4) Запомнить ID заявки (например #2)

Запуск:
```bash
chmod +x race_test.sh
REQUEST_ID=2 ./race_test.sh
```
Windows CMD
```bash
docker compose exec app bash -lc "REQUEST_ID=2 ./race_test.sh"
```
Ожидаемо:
- один запрос: HTTP: 200
- второй запрос: HTTP: 409

Важно: после успешного прогона заявка станет in_progress. Повторный запуск на том же ID
может дать 409/409 — это нормально. Для повторной демонстрации используйте другую assigned
заявку или пересоздайте БД.
